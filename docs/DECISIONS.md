# 설계 결정 기록 (Decision Log)

> promid를 만들며 마주친 핵심 쟁점과, 그때 무엇을 왜 선택했는지 남긴 문서.
> "무슨 기능을 만들었나"가 아니라 **"어떤 트레이드오프에서 왜 이걸 골랐나"**를 기록하는 게 목적이다.
> (각 항목: 배경 → 선택지 → 결정 → 이유)

---

## 1. 출발지 입력 — 텍스트 / 좌표를 한 모델에 (nullable 3필드)

- **배경**: 출발지를 "강남역" 같은 텍스트로도, '현재 위치' 버튼의 좌표로도 받아야 한다.
- **선택지**: (a) 입력 종류별로 DTO 분리, (b) `place / lat / lng` 셋 다 nullable인 한 DTO + "셋 중 최소 하나" 교차 검증.
- **결정**: (b). `StartRequest`에 `@AssertTrue isInputProvided()`로 "place 또는 좌표 중 하나 필수"를 강제.
- **이유**: 같은 "출발지"라는 개념을 DTO 두 개로 쪼개면 컨트롤러·서비스가 분기 지옥이 된다. 한 모델로 받고 검증으로 제약을 거는 게 호출부가 단순.

## 2. 텍스트 → 좌표 변환을 "지오코딩"이 아니라 "키워드 검색"으로

- **배경**: "여의도역 7번출구", "강남역", 일반 주소까지 다 좌표로 바꿔야 한다.
- **선택지**: (a) 주소 지오코딩 API, (b) Kakao 키워드 장소 검색.
- **결정**: (b) 키워드 검색.
- **이유**: 지오코딩은 정형 주소에 강하지만 "역·건물명·출구" 같은 구어 입력엔 약하다. 키워드 검색은 역·건물·주소를 다 처리해서 사용자가 실제로 칠 법한 입력을 폭넓게 커버.

## 3. weight / radius = 선택 + 기본값

- **배경**: 비율(예 7:3)과 검색 반경을 조정 가능하게 하되, 대부분은 기본값으로 충분.
- **결정**: weight 기본 1(균등), radius 기본 1000m. record 압축 생성자에서 null→기본값. weight는 상대 비율이라 정수.
- **이유**: "조정할 수 있지만 신경 안 써도 되는" UX. 필수로 받으면 진입 장벽만 올라간다.

## 4. RestClient — 정적 빌더 vs 빈 주입 (버그에서 배운 것)

- **배경**: Kakao 호출용 `RestClient` 빈을 만들 때 `RestClient.Builder`를 주입하려 했더니 기동 실패.
- **선택지**: (a) `RestClient.Builder` 빈 주입, (b) 정적 `RestClient.builder()`.
- **결정**: (b). `RestClient.builder().baseUrl(...).defaultHeader(...).build()`.
- **이유**: 이 부트 구성에선 `RestClient.Builder` 빈이 자동 구성되지 않아 주입 자체가 불가능했다. IDE의 "autowire 안 됨" 경고를 "무시해도 된다"고 처음에 잘못 판단했던 지점 — **경고를 가볍게 넘기지 말 것**이라는 교훈.

## 5. 테스트 — 나중에 한 번에 vs 기능마다 조금씩

- **배경**: 검증이 전부 수동 curl이었다. 테스트를 언제 도입할지.
- **결정**: 기능 끝낼 때마다 조금씩. 첫 대상은 외부 의존성 없는 순수 로직 `MidpointService`.
- **이유**: "나중에 한 번에"는 양이 부담돼 현실에선 "영원히 안 함"이 된다. 또 코드가 머릿속에 생생할 때 짜야 검증 포인트가 명확. **모든 걸 테스트하지 않고** 가성비 높은 곳(입출력 명확한 순수 로직)부터 — 외부 API·화면은 가성비가 낮아 후순위.

## 6. 자동완성 — 좌표를 "선택하는 순간" 확정

- **배경**: "강남" 검색 시 "강남역사거리 카페" 같은 엉뚱한 첫 결과가 잡혀도 사용자가 알 수 없었다(제출 시점에 백엔드가 첫 결과를 임의 선택).
- **선택지**: (a) 후보 이름만 보여주고 제출 때 다시 지오코딩, (b) 후보 클릭 순간 그 좌표를 프론트가 쥐고, 제출 땐 텍스트가 아니라 lat/lng를 전송.
- **결정**: (b).
- **이유**: 좌표를 **선택 시점에 확정**하면 제출 시점의 모호함이 0이 된다. "내가 고른 그 장소"가 보장됨. 마침 '현재 위치' 버튼이 쓰던 좌표 채우기 메커니즘에 그대로 얹혀서 모델도 안 늘어남.

## 7. 검색 결과 정렬 — Kakao는 "앞글자 자동완성"이 아니다

- **배경**: "강" 한 글자 검색 시 한강·태화강 등 강(river)만 나옴. "앞글자부터 매칭"하는 자동완성을 기대했는데 어긋남.
- **원인**: Kakao 키워드 검색은 사전 기반 prefix typeahead가 아니라 **정확도·인기도순 장소 전체 검색**. 1글자에선 노이즈만.
- **결정**: (1) 받아온 후보를 "이름이 검색어로 시작 > 포함 > 나머지" 순으로 **클라이언트 재정렬**, (2) **최소 2글자**부터 검색.
- **이유**: API 동작을 바꿀 수 없으니, 표시 단에서 직관(이름 매칭 우선)에 맞췄다. 1글자는 의미 있는 후보가 없어 검색 자체를 막는 게 표준 패턴.

## 8. 빈 결과 처리 — 예외 vs 빈 리스트 (맥락에 따라 다름)

- **배경**: Kakao 검색 결과가 0개일 때 어떻게 할지.
- **결정**: `geocode`(제출 시 좌표 확정)는 **예외**, `searchPlaces`(자동완성)는 **빈 리스트**.
- **이유**: 같은 "결과 없음"이라도 의미가 다르다. 제출 시점에 못 찾으면 진짜 에러(예외→400). 자동완성 중 결과 0개는 *정상 상황*(한 글자씩 치는 중) — 여기서 예외를 던지면 에러 배너가 깜빡인다.

## 9. GET + RequestParam vs POST + RequestBody

- **배경**: 검색 엔드포인트를 어떤 형태로 둘지.
- **결정**: 검색은 `GET /api/places/search?query=` (단순 조회·값 하나), 중점 추천은 `POST /api/midpoint` (복잡한 객체·처리).
- **이유**: HTTP 관례 — 조회는 GET, 단순 값은 쿼리 파라미터, 복잡한 객체는 body(JSON). GET은 body를 사실상 못 써서 자연히 RequestParam.

## 10. 공평함의 척도 — 거리 중점 vs 대중교통 시간 (2층 구조로 통합)

- **배경**: "중간에서 만나자"의 공평함을 거리로 볼지 시간으로 볼지. 둘 중 하나를 골라야 하는 것처럼 보였다.
- **핵심 통찰**: 사람이 체감하는 부담은 km가 아니라 **시간**이다(환승·배차 때문에 거리와 시간은 자주 어긋남). 그러나 시간 "중점 좌표"를 직접 구하면 길찾기 API 호출이 폭증하고, 결과가 실시간 지연에 흔들리며, "왜 여기가 중간이냐"를 설명하기 어렵다.
- **결정**: 양자택일이 아니라 **2층 구조**. 거리 중점 = **탐색**(어느 동네서 만날지, 싸고 안정적·결정적), 대중교통 시간 = **정렬**(그 동네 후보 역 ~15개를 공평성으로 순위). 후보가 유한해서 API 호출도 유한(사람 수 × 후보 수).
- **확정**:
  - 길찾기 API = **ODsay LAB** (한국 대중교통 길찾기 표준, 일 1,000건 무료). Kakao/Google/TMAP 대비 "두 좌표 간 대중교통 소요시간"을 무료로 정확히 줌.
  - 공평 기준 = **minimax** (가장 오래 걸리는 사람의 시간을 최소화 → 최악자 보호). 격차·합계 최소화는 보조 지표로 후순위.
  - 채점 후보 수 = **거리순 상위 8개만**. 호출 수 = 후보 수 × 사람 수라, 전체(15개) 채점은 무료 한도·지연 부담. 8개면 중점 근처라 충분.
- **상태**: 방향·API·기준·후보 수 확정. 구현은 로드맵 3단계(임무 9~)에서.
